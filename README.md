# Лабораторная работа  
## Splunk SOC Lab: разработка и валидация правил детектирования для Windows-событий

### Тип работы
Практическая лабораторная работа с последующим оформлением отчёта.

### Цель работы
Развернуть лабораторный стенд на базе **Splunk Enterprise** и подготовить набор базовых **правил детектирования** для событий Windows, имитирующих реальные задачи SOC-аналитика: сбор телеметрии, анализ событий, разработка поисковых запросов в **SPL**, валидация срабатываний и документирование результатов.

---

## Контекст работы
В рамках лабораторной необходимо реализовать небольшой **detection engineering / SOC monitoring** проект.  
Работа должна показать умение:

- развернуть SIEM-стенд;
- подключить источники телеметрии;
- анализировать события безопасности;
- писать запросы на SPL;
- разрабатывать и проверять правила корреляции;
- связывать наблюдаемую активность с MITRE ATT&CK;
- оформлять результаты в виде отчёта, пригодного для демонстрации работодателю.

---

## Ожидаемый результат
По завершении лабораторной работы студент должен получить:

1. Развёрнутый стенд со Splunk;
2. Подключённые журналы Windows Security / Sysmon / PowerShell;
3. Не менее **трёх** реализованных detection use cases;
4. Сохранённые поисковые запросы и/или alert rules;
5. Скриншоты работы стенда;
6. Итоговый отчёт по лабораторной работе.

## Обязательные этапы выполнения

# 1. Подготовка стенда
Необходимо:

- развернуть Splunk Enterprise;
- подготовить Windows-хост как источник событий;
- установить и настроить Splunk Universal Forwarder **или** иной допустимый механизм доставки логов;
- проверить поступление событий в Splunk.

### Что нужно зафиксировать в отчёте
- схема стенда;
- IP-адреса или логические роли узлов;
- какие источники подключены;
- какие sourcetype/index используются.

---

# 2. Настройка журналирования Windows
Необходимо обеспечить наличие телеметрии, пригодной для детектирования атак.

### Минимум:
- Security Events
- Sysmon
- PowerShell logging

### Желательно:
- Sysmon с внятной конфигурацией
- включённый Script Block Logging
- логирование сетевой активности процессов

### Что нужно зафиксировать в отчёте
- какие журналы включены;
- каким образом включались;
- какие Event ID были использованы далее в работе.

---

# 3. Проверка поступления логов в Splunk
До начала детектирования необходимо убедиться, что данные поступают корректно.

### Требуется:
- выполнить базовые поисковые запросы;
- убедиться, что события видны по каждому источнику;
- зафиксировать наличие хотя бы нескольких событий каждого нужного типа.

### Что нужно показать в отчёте
- примеры поисковых запросов;
- скриншоты результатов;
- краткий вывод о корректности ingestion.

---

# 4. Реализация сценариев детектирования
Необходимо реализовать **не менее трёх** detection use cases.

## Обязательные сценарии

### Сценарий 1. Подбор пароля / серия неуспешных входов с последующим успешным входом
**Пример техники:** brute force / password spraying

Нужно:
- определить события неуспешной и успешной аутентификации;
- реализовать правило поиска подозрительной последовательности;
- показать логику корреляции по времени, пользователю, IP-адресу или хосту.

### Сценарий 2. Запуск PowerShell с параметром encoded command
Нужно:
- определить события создания процесса;
- реализовать детект запуска PowerShell с признаками обфускации;
- показать строку запуска процесса.

### Сценарий 3. Подозрительная сетевая активность PowerShell
Нужно:
- зафиксировать команды загрузки удалённого содержимого;
- реализовать поиск по признакам suspicious download activity;
- по возможности связать активность процесса и сетевое соединение.

---

## Допустимые дополнительные сценарии
Дополнительно можно реализовать один или несколько из следующих сценариев:

- запуск `cmd.exe`, `powershell.exe`, `rundll32.exe`, `regsvr32.exe` из нетипичного контекста;
- запуск процессов из временных директорий;
- создание новых локальных пользователей;
- отключение защитных механизмов;
- необычные сетевые соединения;
- множественные неуспешные подключения по RDP;
- execution chain: Office → PowerShell / cmd;
- иные SOC use cases, обоснованные студентом.

---

# 5. Разработка поисковых запросов SPL
Для каждого реализованного сценария необходимо оформить:

1. **Название детекта**
2. **Назначение**
3. **Источники логов**
4. **Ключевые поля**
5. **Поисковый запрос SPL**
6. **Логику срабатывания**
7. **Severity**
8. **MITRE ATT&CK mapping**
9. **Возможные false positives**
10. **Ограничения правила**

---

# 6. Валидация детектов
Каждый детект должен быть **проверен на практике**.

### Требуется:
- сгенерировать тестовые события в лабораторной среде;
- добиться появления релевантных логов в Splunk;
- убедиться, что запрос/правило действительно обнаруживает нужную активность;
- зафиксировать результат.

### Важно
Действия должны выполняться **только в изолированной лабораторной среде**.

---

# 7. Создание артефактов в Splunk
Необходимо подготовить не менее одного из следующих вариантов:

- **saved search**
- **alert**
- **dashboard**
- **report panel**

### Минимальные требования
Должен быть создан как минимум:
- один dashboard **или**
- два сохранённых поиска / alert rules

### Что желательно показать
- панель с динамикой срабатываний;
- панель с подозрительными PowerShell-запусками;
- панель с failed / successful logon activity.

---

# 8. Аналитический разбор результатов
По итогам выполнения лабораторной необходимо проанализировать:

- какие события реально удалось собрать;
- какие детекты сработали устойчиво;
- какие были ложные срабатывания;
- чего не хватает для более зрелого detection engineering;
- как можно улучшить лабораторию.

### 3. Исходные данные и схема стенда

- Все SPL-запросы должны быть приведены в читаемом виде;
- Желательно использовать единый стиль заголовков и нумерации;
- Если использовались сторонние конфигурации или тестовые сценарии, это необходимо указать.

## Критерии оценки

### Оценка "удовлетворительно"
- стенд развёрнут;
- логи поступают;
- реализован минимум один рабочий детект;
- отчёт оформлен частично.

### Оценка "хорошо"
- реализованы три сценария;
- есть валидные SPL-запросы;
- приведены скриншоты и пояснения;
- есть базовый анализ результатов.

### Оценка "отлично"
- реализованы все обязательные сценарии;
- детекты валидированы;
- есть dashboard / alerts / saved searches;
- выполнен MITRE ATT&CK mapping;
- описаны false positives и пути улучшения;
- отчёт выглядит как практический кейс для портфолио SOC-аналитика.
